Frontend Overhaul — 3 Approaches

Pick the approach that best matches your goals for Vision GridAI as a sellable SaaS product

A

Evolutionary Rebuild — Keep Vite + React Router

Stack: Vite + React 18 + React Router 7 + shadcn/ui + Tailwind v4 + React Query

What changes: Replace hand-rolled components with shadcn/ui. New Figma-inspired design system. Rebuild all 12 pages. Keep current routing and data layer intact.

Pros

  • Fastest to ship — existing hooks, API layer, and Supabase Realtime all stay
  • Zero migration risk — no new framework to learn
  • shadcn/ui is copy-paste, not a dependency — full control
  • Vite is the fastest dev server, perfect for SPA

Cons

  • No SSR — not ideal if you later want SEO or public-facing pages
  • Client-side routing only — no server-side data loading

⭐ Recommended. Your dashboard is a private SPA behind auth — no SEO needed. The existing data layer (React Query + Supabase Realtime) is already well-built. This gives you the biggest visual upgrade with the least risk.

B

Next.js Migration — Full Framework Upgrade

Stack: Next.js 15 (App Router) + shadcn/ui + Tailwind v4 + Server Components

What changes: Migrate to Next.js. Server components for data loading. File-based routing. API routes replace n8n webhook proxying. Full rebuild.

Pros

  • Industry standard for SaaS — investors/buyers recognize it
  • Server components for faster initial loads
  • Built-in API routes, middleware, auth patterns
  • Easy to add public marketing pages later

Cons

  • Major migration effort — every page, hook, and route changes
  • Supabase Realtime requires client components — limits Server Component benefits
  • Needs Node.js server (not just static files on Nginx)
  • Your dashboard is private SPA — SSR benefits are minimal
  • 2-3x longer to ship vs Approach A
C

Hybrid — Vite SPA Now, Next.js Later

Stack: Vite + React 18 + shadcn/ui now. Designed so pages can migrate to Next.js incrementally.

What changes: Same as Approach A, but with architectural patterns (barrel exports, co-located data loaders) that make a future Next.js migration easy.

Pros

  • Ships fast like A, migrates like B later
  • No wasted work — components and hooks transfer 1:1

Cons

  • Premature abstraction risk — designing for a migration that may never happen
  • Slightly more complex patterns for no immediate benefit
