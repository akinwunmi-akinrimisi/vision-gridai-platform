# INFR-04: Dashboard Deploy to Nginx

## Goal

Build the React dashboard on the VPS and serve it via Nginx with HTTPS at `https://dashboard.operscale.cloud`.

## must_haves

- [ ] `https://dashboard.operscale.cloud` responds in a browser with the React SPA
- [ ] SSL certificate is valid (Let's Encrypt / Certbot)
- [ ] Nginx config is installed at `/etc/nginx/sites-available/vision-gridai` and symlinked to `sites-enabled`
- [ ] React build output is at `/opt/dashboard/index.html`
- [ ] `chown -R www-data:www-data /opt/dashboard` applied
- [ ] `dashboard/.env` on VPS contains `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` before build

## Context

- Build on VPS (not local + SCP) — SSH in, git pull, npm install, npm run build
- Repo on VPS: `/root/vision-gridai-platform`
- Build output: `dashboard/dist/` → deployed to `/opt/dashboard/`
- Nginx root: `/opt/dashboard`
- Dashboard URL: `https://dashboard.operscale.cloud`
- SSL: Certbot Let's Encrypt (same pattern as n8n on this VPS)
- Nginx config template: `dashboard/nginx.conf` (already written — HTTP only, needs HTTPS upgrade)
- VITE env vars: `dashboard/src/lib/supabase.js` reads `import.meta.env.VITE_SUPABASE_URL` and `import.meta.env.VITE_SUPABASE_ANON_KEY` — must be in `dashboard/.env` before `npm run build`
- VPS SSH: `ssh root@srv1297445.hstgr.cloud`

## Tasks

<task id="4.1" name="Pre-flight checks on VPS (nginx, node, dns, disk)">

SSH into the VPS:
```bash
ssh root@srv1297445.hstgr.cloud
```

Run all pre-flight checks:

**Check 1: Nginx installed**
```bash
which nginx
```
If missing: `apt update && apt install -y nginx`

**Check 2: Node.js installed**
```bash
node --version
```
If missing, install via nvm (preferred) or apt:
```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
source ~/.bashrc
nvm install --lts
```

**Check 3: Disk space**
```bash
df -h /root
```
Must show at least 2GB free. `npm install` for the dashboard needs ~300MB, `node_modules` cache grows over time. If less than 2GB free, clean docker images first:
```bash
docker image prune -f
```

**Check 4: DNS resolution**
```bash
dig dashboard.operscale.cloud +short
```
Must return the VPS IP (not empty). If empty, DNS A record for `dashboard.operscale.cloud` is not set — surface this to the user: "Add an A record for dashboard.operscale.cloud pointing to <VPS IP> in your DNS provider, then re-run." Do not proceed with Certbot until DNS resolves.

**Check 5: Repo present on VPS**
```bash
ls /root/vision-gridai-platform 2>/dev/null && echo "EXISTS" || echo "MISSING"
```
If MISSING:
```bash
git clone https://github.com/<your-repo>/vision-gridai-platform.git /root/vision-gridai-platform
```
If EXISTS:
```bash
cd /root/vision-gridai-platform && git pull origin main
```
</task>

<task id="4.2" name="Ensure dashboard/.env exists on VPS with VITE_ vars" depends_on="4.1">

The React build bakes environment variables into the JS bundle at build time. `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` must exist in `dashboard/.env` before `npm run build`.

On the VPS:
```bash
ls /root/vision-gridai-platform/dashboard/.env 2>/dev/null && echo "EXISTS" || echo "MISSING"
```

If EXISTS, verify it contains the VITE_ vars:
```bash
grep "VITE_SUPABASE" /root/vision-gridai-platform/dashboard/.env
```
Expected: two lines with `VITE_SUPABASE_URL=...` and `VITE_SUPABASE_ANON_KEY=...`.

If MISSING or incomplete, create/update it:
```bash
cat > /root/vision-gridai-platform/dashboard/.env << 'EOF'
VITE_SUPABASE_URL=https://supabase.operscale.cloud
VITE_SUPABASE_ANON_KEY=<paste-anon-key-here>
EOF
```

Replace `<paste-anon-key-here>` with the actual Supabase anon key. The anon key can be found:
- In the root `.env` file of the repo (if committed — check `dashboard/.env` in git)
- In the Supabase dashboard at `https://supabase.operscale.cloud` → Project Settings → API

Note: `dashboard/.env` is gitignored. If it does not exist on the VPS, it must be created manually with the correct values. This is the only step in this plan requiring a secret.
</task>

<task id="4.3" name="Build the React dashboard on the VPS" depends_on="4.2">

```bash
cd /root/vision-gridai-platform/dashboard
npm install
npm run build
```

`npm install` downloads dependencies to `node_modules/` (~300MB).
`npm run build` runs Vite and outputs to `dashboard/dist/`.

Verify build succeeded:
```bash
ls /root/vision-gridai-platform/dashboard/dist/index.html
```

If `index.html` is present, build succeeded. If the build fails, check:
- Node.js version (Vite requires Node 18+): `node --version`
- Missing VITE_ vars will NOT cause a build failure — they'll be baked in as `undefined`, causing runtime errors. This is why task 4.2 must complete first.

Estimated build time: 30–90 seconds.
</task>

<task id="4.4" name="Deploy React build to /opt/dashboard" depends_on="4.3">

```bash
mkdir -p /opt/dashboard
rm -rf /opt/dashboard/*
cp -r /root/vision-gridai-platform/dashboard/dist/. /opt/dashboard/
chown -R www-data:www-data /opt/dashboard
```

Verify deployment:
```bash
ls /opt/dashboard/index.html
ls /opt/dashboard/assets/
```

Both must exist. The `assets/` directory contains the hashed JS and CSS bundles.
</task>

<task id="4.5" name="Install Nginx config and enable HTTPS with Certbot" depends_on="4.4">

**Step 1: Update nginx.conf in the repo to include HTTPS redirect**

The existing `dashboard/nginx.conf` serves HTTP only. Certbot will manage HTTPS injection, but we need to start with the HTTP config installed so Certbot can verify domain ownership via HTTP-01 challenge.

Install the HTTP config first:
```bash
cp /root/vision-gridai-platform/dashboard/nginx.conf /etc/nginx/sites-available/vision-gridai
ln -sf /etc/nginx/sites-available/vision-gridai /etc/nginx/sites-enabled/vision-gridai
```

Remove the default Nginx site if present (to avoid port 80 conflict):
```bash
rm -f /etc/nginx/sites-enabled/default
```

Test and reload Nginx:
```bash
nginx -t && systemctl reload nginx
```

**Step 2: Obtain SSL certificate via Certbot**

Install Certbot if not present:
```bash
which certbot || apt install -y certbot python3-certbot-nginx
```

Run Certbot (it will automatically detect the Nginx config and add HTTPS):
```bash
certbot --nginx -d dashboard.operscale.cloud --non-interactive --agree-tos --email admin@operscale.cloud --redirect
```

The `--redirect` flag makes Certbot add an HTTP→HTTPS redirect automatically. The `--nginx` flag tells Certbot to modify the Nginx config directly rather than using webroot mode.

After Certbot completes, verify the Nginx config was updated:
```bash
grep "ssl_certificate" /etc/nginx/sites-available/vision-gridai
```
Expected: lines with `/etc/letsencrypt/live/dashboard.operscale.cloud/fullchain.pem` and `privkey.pem`.

**Step 3: Final Nginx reload**
```bash
nginx -t && systemctl reload nginx
```

If Certbot fails due to DNS not resolving, surface the error: DNS A record is missing. Retry after DNS propagation.

Note (per CONTEXT.md Claude's Discretion): Nginx config file is named `vision-gridai` and uses `sites-available` + symlink pattern.
</task>

<task id="4.6" name="Smoke test: verify dashboard loads at production URL" depends_on="4.5">

On the VPS, use curl to verify HTTPS is working:
```bash
curl -s -o /dev/null -w "%{http_code}" https://dashboard.operscale.cloud
```

Expected: `200`

If `301` or `302`: follow the redirect:
```bash
curl -sL -o /dev/null -w "%{http_code}" https://dashboard.operscale.cloud
```

If `502`: Nginx is running but something in the config is broken. Check:
```bash
nginx -t
systemctl status nginx
```

If `curl: (60)` SSL error: certificate wasn't issued correctly — rerun certbot:
```bash
certbot --nginx -d dashboard.operscale.cloud --non-interactive --agree-tos --email admin@operscale.cloud --redirect
```

A `200` response confirms Nginx is serving the React bundle. The final verification (visual UI check) requires browser access per CONTEXT.md.
</task>
