## Context

Current `infra/docker-compose.override.yml` has conservative limits for a 4GB VPS. The production VPS has 16GB RAM. Updating to:
- n8n: 4GB (was 1536M)
- supabase-db: 4GB (was 1024M)
- supabase-realtime: 1GB (was 512M)
- supabase-rest: 512MB (was 256M)

PostgreSQL NVMe tuning (new values):
- shared_buffers: 1GB (was 256MB)
- effective_cache_size: 3GB (was 512MB)
- random_page_cost: 1.1 (new — NVMe SSDs have near-random I/O)
- shm_size: 256m at service level (required for shared_buffers=1GB to work in Docker)

n8n env vars (adding/updating):
- N8N_PAYLOAD_SIZE_MAX=256 (new — was missing)
- N8N_BINARY_DATA_TTL=168 (new — 7-day binary data TTL for pruning)
- NODE_OPTIONS=--max-old-space-size=2048 (update from 1024)
- EXECUTIONS_TIMEOUT=600 (already present — preserve it)

Workflow: update file locally → commit → push → SSH to VPS → git pull → docker compose restart.

## Tasks

<task id="1.1" name="Rewrite infra/docker-compose.override.yml with all Wave 1 changes">

Overwrite `infra/docker-compose.override.yml` with the following complete content. This single file update incorporates INFR-01 (memory limits), INFR-02 (PG tuning), and INFR-03 (n8n env vars) so all three can be applied in one docker compose restart:

```yaml
# ================================================================
# Docker Compose Override for Vision GridAI
# ================================================================
# Usage: docker compose -f docker-compose.yml -f docker-compose.override.yml up -d
#
# Tuned for 16GB VPS (Hostinger KVM). Updated 2026-03-09.
#
# Total memory allocated: ~9.5GB (leaves headroom for OS + Nginx + buffers)
#   n8n:              4096M (Node.js + workflow execution + 2GB heap)
#   supabase-db:      4096M (PostgreSQL: 1GB shared_buffers + connections + query cache)
#   supabase-realtime: 1024M (WebSocket connections from dashboard)
#   supabase-rest:     512M  (PostgREST API layer)
# ================================================================

version: "3.8"

services:
  # n8n container -- needs memory for Node.js + workflow execution
  # Long-running workflows (script gen, niche research) need generous timeouts
  n8n:
    deploy:
      resources:
        limits:
          memory: 4096M
        reservations:
          memory: 1024M
    environment:
      - EXECUTIONS_TIMEOUT=600
      - EXECUTIONS_TIMEOUT_MAX=900
      - N8N_EXECUTIONS_PROCESS_TIMEOUT=600
      - N8N_DEFAULT_BINARY_DATA_MODE=filesystem
      - EXECUTIONS_DATA_PRUNE=true
      - EXECUTIONS_DATA_MAX_AGE=168
      - NODE_OPTIONS=--max-old-space-size=2048
      - N8N_PAYLOAD_SIZE_MAX=256
      - N8N_BINARY_DATA_TTL=168

  # Supabase PostgreSQL -- NVMe-tuned for production workloads
  # shm_size required for shared_buffers >= 256MB in Docker
  # Command-line flags override postgresql.conf without editing the file
  supabase-db:
    shm_size: 256m
    deploy:
      resources:
        limits:
          memory: 4096M
        reservations:
          memory: 1024M
    command: >
      postgres
      -c max_connections=200
      -c shared_buffers=1GB
      -c work_mem=8MB
      -c effective_cache_size=3GB
      -c random_page_cost=1.1
      -c statement_timeout=30s
      -c idle_in_transaction_session_timeout=300s

  # Supabase Realtime -- WebSocket connections from dashboard
  # Each dashboard tab holds 2-3 subscriptions (scenes, topics, projects)
  supabase-realtime:
    deploy:
      resources:
        limits:
          memory: 1024M
        reservations:
          memory: 256M

  # Supabase PostgREST -- REST API layer
  # Lightweight: just translates HTTP to SQL
  supabase-rest:
    deploy:
      resources:
        limits:
          memory: 512M
        reservations:
          memory: 128M
```

After writing, verify the file reads back correctly:
```bash

```

Confirm all key values are present: `4096M`, `shm_size: 256m`, `shared_buffers=1GB`, `effective_cache_size=3GB`, `random_page_cost=1.1`, `N8N_PAYLOAD_SIZE_MAX=256`, `max-old-space-size=2048`.
</task>
