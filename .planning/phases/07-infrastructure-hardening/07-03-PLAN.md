# INFR-03: n8n Environment Variables Verification

## Goal

Verify that the `n8n-ffmpeg` container is running with the correct environment variables applied by plan 07-01: updated `NODE_OPTIONS` (2GB heap), new `N8N_PAYLOAD_SIZE_MAX=256` and `N8N_BINARY_DATA_TTL=168`, and preserved `EXECUTIONS_TIMEOUT=600`.

## must_haves

- [ ] `NODE_OPTIONS=--max-old-space-size=2048` is set in the n8n container (was 1024)
- [ ] `N8N_PAYLOAD_SIZE_MAX=256` is set in the n8n container (was missing)
- [ ] `N8N_BINARY_DATA_TTL=168` is set in the n8n container (was missing)
- [ ] `EXECUTIONS_TIMEOUT=600` is preserved in the n8n container (was already set, must survive update)

## Context

The n8n container name is `n8n-ffmpeg` per CLAUDE.md — this is a custom container image that bundles n8n with FFmpeg for the assembly workflow. The environment variables control:

- `NODE_OPTIONS=--max-old-space-size=2048`: Increases Node.js V8 heap from 1GB to 2GB — prevents OOM crashes during large script generation workflows
- `N8N_PAYLOAD_SIZE_MAX=256`: Allows webhook payloads up to 256MB — needed for large scene manifest JSONs
- `N8N_BINARY_DATA_TTL=168`: Sets binary data TTL to 168 hours (7 days) — completes the pruning configuration already set by `EXECUTIONS_DATA_MAX_AGE=168`
- `EXECUTIONS_TIMEOUT=600`: 10-minute execution timeout — must not be lost in the update (regression check)

VPS SSH: `ssh root@srv1297445.hstgr.cloud`

## Tasks

<task id="3.1" name="SSH to VPS and verify n8n container environment variables">

This task runs after 07-01 has applied the docker compose restart.

SSH into the VPS:
```bash
ssh root@srv1297445.hstgr.cloud
```

Run the environment variable check (exact command from CONTEXT.md):
```bash
docker exec n8n-ffmpeg env | grep -E 'EXECUTIONS|NODE_OPTIONS|N8N_PAYLOAD|N8N_BINARY'
```

Expected output (order may vary):
```
EXECUTIONS_TIMEOUT=600
EXECUTIONS_TIMEOUT_MAX=900
N8N_EXECUTIONS_PROCESS_TIMEOUT=600
EXECUTIONS_DATA_PRUNE=true
EXECUTIONS_DATA_MAX_AGE=168
NODE_OPTIONS=--max-old-space-size=2048
N8N_PAYLOAD_SIZE_MAX=256
N8N_BINARY_DATA_TTL=168
```

Key assertions — ALL must be true:
- `NODE_OPTIONS` must contain `2048`, not `1024`
- `N8N_PAYLOAD_SIZE_MAX=256` must be present
- `N8N_BINARY_DATA_TTL=168` must be present
- `EXECUTIONS_TIMEOUT=600` must be present (regression check)

If `NODE_OPTIONS` still shows `1024`:
```bash
docker inspect n8n-ffmpeg | grep -A 20 '"Env"'
```
Check if the override env vars are in the container's environment list. If not, the container was not restarted with the override — repeat task 1.3 from plan 07-01.

If `n8n-ffmpeg` container is not found:
```bash
docker ps --format '{{.Names}}' | grep -i n8n
```
Use the actual container name if it differs.
</task>

<task id="3.2" name="Verify n8n is responding to webhooks after restart" depends_on="3.1">

After confirming env vars are correct, do a quick health check to ensure n8n is operational (the restart in 07-01 takes n8n briefly offline):

```bash
curl -s -o /dev/null -w "%{http_code}" https://n8n.srv1297445.hstgr.cloud/healthz
```

Expected: `200`

If `000` (connection refused) or non-200, n8n is still starting up. Wait 15 seconds and retry:
```bash
sleep 15 && curl -s -o /dev/null -w "%{http_code}" https://n8n.srv1297445.hstgr.cloud/healthz
```

If n8n fails to start, check logs:
```bash
docker logs n8n-ffmpeg --tail 50
```

Common failure: heap allocation error if `NODE_OPTIONS` exceeds physical memory. The 2GB heap + 4GB container limit is safe for a 16GB VPS with multiple containers.
</task>
