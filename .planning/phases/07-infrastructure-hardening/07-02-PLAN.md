# INFR-02: PostgreSQL NVMe Tuning Verification

## Goal

Verify that the PostgreSQL instance running inside the Supabase Docker container has loaded the NVMe-optimized settings applied by plan 07-01: `shared_buffers=1GB`, `effective_cache_size=3GB`, `random_page_cost=1.1`.

## must_haves

- [ ] `SHOW shared_buffers;` inside the container returns `1GB`
- [ ] `SHOW effective_cache_size;` inside the container returns `3GB`
- [ ] `SHOW random_page_cost;` inside the container returns `1.1`
- [ ] `shm_size: 256m` was applied (container started without IPC errors in logs)

## Context

The PostgreSQL settings are applied via command-line flags in the `docker-compose.override.yml` `command:` block. This approach overrides postgresql.conf values without editing the config file directly — no config file on disk to verify, the running process parameters are the ground truth.

Changes applied by 07-01:
- `shared_buffers=1GB` (was 256MB) — uses ~25% of the 4GB container allocation for PostgreSQL's shared memory
- `effective_cache_size=3GB` (was 512MB) — tells the planner how much OS cache is available, improves query plan selection
- `random_page_cost=1.1` (new) — tells the planner NVMe random reads are almost as cheap as sequential reads
- `shm_size: 256m` at the service level — required for Docker to allow the PostgreSQL process to allocate 1GB shared_buffers

Container name discovery: use `docker ps --format '{{.Names}}' | grep -i postgres` — Supabase's postgres container is typically named `supabase-db` or `supabase_db` or similar.

VPS SSH: `ssh root@srv1297445.hstgr.cloud`

<task id="2.2" name="Verify PostgreSQL loaded NVMe tuning parameters" depends_on="2.1">

Using the container name discovered in task 2.1, run the verification query:

```bash
docker exec <postgres-container-name> psql -U postgres -c 'SHOW shared_buffers; SHOW effective_cache_size; SHOW random_page_cost;'
```

Example (if container is named `supabase-db`):
```bash
docker exec supabase-db psql -U postgres -c 'SHOW shared_buffers; SHOW effective_cache_size; SHOW random_page_cost;'
```

Expected output:
```
 shared_buffers
----------------
 1GB
(1 row)

 effective_cache_size
---------------------
 3GB
(1 row)

 random_page_cost
-----------------
 1.1
(1 row)
```

If any value shows the old defaults (e.g., shared_buffers shows `128MB` or `256MB`), the docker compose override is not being applied correctly. In this case:

1. Verify the container was actually restarted in 07-01 (not just `docker compose restart` without `-f` override flags):
```bash
docker inspect <postgres-container-name> | grep -A 5 "Cmd"
```
Look for `-c shared_buffers=1GB` in the Cmd array. If absent, the container is running from the base compose without the override — repeat task 1.3 from plan 07-01.

2. Check for IPC errors (shared memory allocation failures):
```bash
docker logs <postgres-container-name> --tail 30 | grep -i "shm\|shared"
```
If present, verify `shm_size: 256m` is in the override and was picked up.
</task>
